# Kiro Progressive Steering

Reference for the Progressive Steering surface in Kanon's Kiro adapter — fields, flags, configuration, and audit markers.

---

## Inclusion Modes

Kiro steering files support three inclusion modes that control when a file is loaded into the agent's context:

| Mode | Loaded when… |
|------|--------------|
| `always` | Every agent interaction |
| `fileMatch` | A file matching a glob pattern is in context |
| `manual` | The user explicitly references the file via `#` |

**Progressive Steering** means choosing the lightest mode that still delivers guidance at the right moment — avoiding unnecessary context in every prompt.

---

## `harness-config.kiro.inclusion`

Declare the Kiro inclusion mode for an artifact in its `knowledge.md` frontmatter:

```yaml
harness-config:
  kiro:
    inclusion: fileMatch
    fileMatchPattern: "src/auth/**/*.ts"
```

### Values

| Value | Effect |
|-------|--------|
| `always` | Steering file loaded into every interaction |
| `fileMatch` | Loaded only when an open file matches `fileMatchPattern` |
| `manual` | Loaded only on explicit `#`-reference by the user |

### Precedence

The Kiro adapter resolves the emitted `inclusion` value using this order (highest wins):

1. `harness-config.kiro.inclusion` — harness-specific, recommended
2. Top-level `frontmatter.inclusion` — legacy, shared across harnesses
3. Default: `always` (with an informational warning recommending explicit declaration)

Setting `harness-config.kiro.inclusion` explicitly suppresses the default-inclusion warning.

---

## `harness-config.kiro.fileMatchPattern`

A glob expression emitted as `fileMatchPattern:` in the compiled steering file's YAML frontmatter.

- **Required** when `inclusion: fileMatch`. The validator produces an error if absent or empty.
- **Ignored** when `inclusion` is `always` or `manual`. If present with those modes, the validator emits a warning that the field has no effect.

### Examples

```yaml
harness-config:
  kiro:
    inclusion: fileMatch
    fileMatchPattern: "src/**/*.test.ts"    # fires when test files are open
```

```yaml
harness-config:
  kiro:
    inclusion: fileMatch
    fileMatchPattern: "**/*.sql"            # fires on any SQL file
```

## `--max-always` (install flag)

Limits the number of `always`-mode Kiro steering files that a single `kanon install` invocation can write.

```bash
# Reject any install that adds always-on steering
kanon install my-artifact --harness kiro --max-always 0

# Allow at most 2 always-on files per install
kanon install my-artifact --harness kiro --max-always 2
```

### Behaviour

- When the count of would-be-installed `always` files exceeds the limit, the install **aborts before writing any files** and exits with a non-zero code.
- Files with unparseable or missing `inclusion` frontmatter are conservatively counted as `always`.
- Omitting `--max-always` disables the limit entirely (backward-compatible default).
- Works with `--dry-run`: a breach still produces a non-zero exit so CI pipelines can detect it without modifying the filesystem.

## `kiro.progressiveSteering.alwaysWarnThreshold`

A build-time configuration key in `kanon.config.yaml` that controls when the build emits a threshold warning about always-on context.

```yaml
kiro:
  progressiveSteering:
    alwaysWarnThreshold: 0.5
```

### Details

| Property | Value |
|----------|-------|
| Type | Number (0..1) |
| Default | `0.5` |
| Location | `kanon.config.yaml` at project root |

When the fraction of compiled Kiro steering files with `inclusion: always` exceeds this threshold **and** at least two Kiro steering files are compiled, `kanon build` emits a warning naming each contributing artifact.

- Set to `1` to disable the warning entirely.
- Set to `0` to warn on any `always` file (strictest).
- With `kanon build --strict`, the warning is promoted to an error and the build exits non-zero.

## Audit Comment

Every compiled Kiro steering file includes a single-line HTML comment recording the resolved inclusion mode:

```html
<!-- forge:kiro-inclusion: always -->
```

```html
<!-- forge:kiro-inclusion: fileMatch fileMatchPattern=src/auth/**/*.ts -->
```

```html
<!-- forge:kiro-inclusion: manual -->
```

### Placement

The comment appears immediately after the closing `---` of the frontmatter block and before the `Generated by Kanon` comment. It does not affect Kiro's runtime behaviour — it is a persistent audit marker so consumers can inspect what was chosen without re-parsing frontmatter.

## `kanon eval --harness kiro --rubric progressive-steering`

Run the Progressive Steering quality rubric against a compiled Kiro build.

```bash
# Run against an existing build directory
kanon eval --harness kiro --rubric progressive-steering --build dist/kiro

# Machine-readable JSON for CI
kanon eval --harness kiro --rubric progressive-steering --build dist/kiro --json rubric.json
```

The rubric computes six metrics (AOCW, PR, FMP, MD, DER, WCA), produces a composite score (0–100), and assigns a Green / Yellow / Red rating.

| Exit Code | Meaning |
|-----------|---------|
| 0 | Green or Yellow |
| 1 | Red |

Full metric definitions, thresholds, and workload fixture format: [Kiro Progressive Steering Rubric](kiro-progressive-steering-rubric.md).

## Build & Install Summaries

### Build summary

When `kanon build` compiles Kiro artifacts, it prints an **Inclusion Summary** to stderr:

```
Kiro inclusion summary: 5 files (3 always [2 steering, 1 power], 1 fileMatch, 1 manual) — 40% progressive
```

The summary distinguishes `steering` format from `power` format and reports the progressive ratio.

### Install summary

After `kanon install` writes Kiro files, it prints the same grouped summary for the files just installed. This runs regardless of `--max-always` (the summary shows what landed or would land).
