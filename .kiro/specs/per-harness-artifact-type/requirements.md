# Requirements Document

## Introduction

The Skill Forge tool currently uses a single top-level `type` field in artifact frontmatter (`skill`, `power`, or `rule`) to classify knowledge artifacts. This abstraction is misleading because the concept of artifact "type" is harness-dependent — each AI coding tool has its own native output formats and none of them share the `skill`/`power`/`rule` taxonomy. In practice, no adapter actually reads the top-level `type` field to determine output format; Kiro already uses `harness-config.kiro.power: true` to toggle power behavior independently.

This feature deprecates the global `type` field and lets each harness adapter drive its own output format through the existing `harness-config` section. Each harness defines its own set of valid output formats (e.g., Kiro: `steering` or `power`; Copilot: `instructions` or `agent`; Q Developer: `rule` or `agent`). The wizard is updated to explain what each harness selection means and what output format options are available per harness.

## Glossary

- **Artifact**: A knowledge artifact authored in Skill Forge, consisting of a `knowledge.md` file with YAML frontmatter, optional `hooks.yaml`, optional `mcp-servers.yaml`, and optional workflows.
- **Frontmatter**: The YAML metadata block at the top of `knowledge.md` that defines artifact properties.
- **Harness**: A target AI coding tool (Kiro, Cursor, Copilot, Claude Code, Windsurf, Cline, Q Developer) that Skill Forge compiles artifacts for.
- **Harness_Config**: The `harness-config` section in frontmatter that holds per-harness settings, keyed by harness name.
- **Output_Format**: A harness-native classification that determines what files and directory structure an Adapter produces for a given Harness. Each Harness defines its own set of valid Output_Format values.
- **Adapter**: A harness-specific module in `src/adapters/` that transforms a canonical KnowledgeArtifact into output files for a particular Harness.
- **Wizard**: The interactive CLI prompt flow in `src/wizard.ts` that guides users through creating a new artifact.
- **Catalog**: The JSON index of all artifacts generated by `src/catalog.ts`, consumed by the Browse SPA.
- **Browse_SPA**: The single-page web application served by `src/browse.ts` that displays the Catalog with search and filter capabilities.
- **FrontmatterSchema**: The Zod validation schema in `src/schemas.ts` that defines the structure and defaults for artifact frontmatter.
- **ArtifactTypeSchema**: The current Zod enum schema `z.enum(["skill", "power", "rule"])` in `src/schemas.ts`, to be deprecated.

## Requirements

### Requirement 1: Deprecate Global Type Field

**User Story:** As an artifact author, I want the system to stop requiring a single global `type` classification, so that I am not forced to pick a label that may not apply to all my target harnesses.

#### Acceptance Criteria

1. THE FrontmatterSchema SHALL retain the top-level `type` field as an optional field with a default value of `skill` for backward compatibility.
2. THE system SHALL emit a deprecation warning during `forge validate` when an artifact relies on the top-level `type` field without per-harness output format configuration.
3. THE Wizard SHALL no longer prompt for a global artifact type as a required step.
4. THE top-level `type` field SHALL NOT influence adapter output; adapters SHALL determine output format exclusively from their Harness_Config section.

### Requirement 2: Per-Harness Output Format in Harness Config

**User Story:** As an artifact author, I want to specify the output format for each harness in the `harness-config` section using that harness's native terminology, so that I can control how my artifact is compiled for each target tool.

#### Acceptance Criteria

1. THE Harness_Config sub-schema for each Harness SHALL accept an optional `format` field whose valid values are defined per-harness.
2. THE valid `format` values for each Harness SHALL be:
   - **kiro**: `steering` (default) or `power` — where `steering` produces a steering markdown file and `power` produces POWER.md with a steering subdirectory
   - **cursor**: `rule` (default) — produces `.cursor/rules/` files; Cursor has a single output format
   - **copilot**: `instructions` (default) or `agent` — where `instructions` produces `copilot-instructions.md` and `agent` produces `AGENTS.md`
   - **claude-code**: `claude-md` (default) — produces `CLAUDE.md`; Claude Code has a single output format
   - **windsurf**: `rule` (default) — produces `.windsurf/rules/` files
   - **cline**: `rule` (default) — produces `.clinerules/` files
   - **qdeveloper**: `rule` (default) or `agent` — where `rule` produces `.q/rules/` files and `agent` produces `.q/agents/` files
3. WHEN a Harness_Config section omits the `format` field, THE Adapter SHALL use the default Output_Format for that Harness.
4. WHEN a Harness_Config section contains a `format` value not in the Harness's valid set, THE FrontmatterSchema SHALL reject the frontmatter with a descriptive validation error identifying the Harness and the invalid value.

### Requirement 3: Adapter Output Format Resolution

**User Story:** As a build pipeline consumer, I want each harness adapter to read its output format from harness-config and produce the corresponding native output, so that compiled artifacts match the author's intent per harness.

#### Acceptance Criteria

1. WHEN building output for a Harness, THE Adapter SHALL read the `format` field from that Harness's Harness_Config section.
2. WHEN no `format` field exists for a Harness, THE Adapter SHALL use the default Output_Format for that Harness.
3. THE Kiro Adapter SHALL produce power-specific output (POWER.md, steering subdirectory) when `format` is `power`, and standard steering output when `format` is `steering`.
4. THE Copilot Adapter SHALL produce `AGENTS.md` output when `format` is `agent`, and `copilot-instructions.md` output when `format` is `instructions`.
5. THE Q Developer Adapter SHALL produce `.q/agents/` output when `format` is `agent`, and `.q/rules/` output when `format` is `rule`.
6. THE Kiro Adapter SHALL migrate from reading `harness-config.kiro.power: true` to reading `harness-config.kiro.format: power`, with backward compatibility for the existing `power: true` flag during a deprecation period.

### Requirement 4: Wizard Harness Context and Format Prompting

**User Story:** As an artifact author using the interactive wizard, I want the wizard to explain what each harness selection means and let me choose the output format per harness, so that I understand the significance of my choices.

#### Acceptance Criteria

1. THE Wizard's harness multi-select prompt SHALL display a brief description for each harness explaining what it is and what output it produces (e.g., "kiro — Generates steering files or powers for Kiro IDE").
2. AFTER the user selects harnesses, THE Wizard SHALL prompt for the output format for each selected harness that supports multiple formats.
3. THE Wizard SHALL skip the format prompt for harnesses that have only one output format (Cursor, Claude Code, Windsurf, Cline).
4. WHEN prompting for a harness's output format, THE Wizard SHALL display the valid format options with descriptions explaining what each format produces (e.g., "steering — A markdown knowledge file included in AI context" vs "power — A capability bundle with POWER.md and steering directory").
5. THE Wizard SHALL write the selected format values into the appropriate Harness_Config sections in the generated frontmatter.
6. WHEN the user selects the default format for all harnesses, THE Wizard SHALL omit the `format` field from Harness_Config to keep the frontmatter minimal.

### Requirement 5: Catalog Output Format Representation

**User Story:** As a catalog consumer, I want the catalog to represent the output format on a per-harness basis, so that tools and UIs can display what each artifact produces for each harness.

#### Acceptance Criteria

1. THE CatalogEntrySchema SHALL include a `formatByHarness` field that maps each Harness in the artifact's `harnesses` array to its resolved Output_Format string.
2. THE CatalogEntrySchema SHALL retain the top-level `type` field for backward compatibility, populated from the deprecated frontmatter `type` field.
3. WHEN generating a catalog entry, THE Catalog generator SHALL populate `formatByHarness` by reading each Harness's Harness_Config `format` field, falling back to the Harness's default Output_Format.

### Requirement 6: Browse SPA Format Display

**User Story:** As a catalog browser user, I want the browse SPA to show what output format each artifact produces per harness, so that I can understand what I'll get when I install it.

#### Acceptance Criteria

1. THE Browse_SPA card view SHALL display the per-harness output formats as compact labels alongside the harness names (e.g., "kiro:power", "cursor:rule").
2. THE Browse_SPA detail view SHALL display the full `formatByHarness` mapping with descriptions of what each format produces.
3. THE Browse_SPA type filter SHALL be replaced with a format filter that matches artifacts where at least one harness uses the selected format.

### Requirement 7: Backward Compatibility

**User Story:** As an existing artifact author, I want my current artifacts to continue working without modification, so that the migration path is zero-effort.

#### Acceptance Criteria

1. WHEN an artifact's frontmatter contains a top-level `type` field and no `format` in any Harness_Config section, THE system SHALL behave identically to the current behavior — each Adapter uses its default Output_Format.
2. WHEN an artifact's frontmatter omits the `type` field entirely, THE FrontmatterSchema SHALL default the type to `skill`, consistent with current behavior.
3. WHEN the Kiro Adapter encounters `harness-config.kiro.power: true` without a `format` field, THE Kiro Adapter SHALL treat it as `format: power` for backward compatibility and emit a deprecation notice suggesting migration to `format: power`.
4. THE Catalog generator SHALL produce valid catalog entries for artifacts that use only the top-level `type` field, with `formatByHarness` populated from each Harness's default Output_Format.

### Requirement 8: Schema Validation for Per-Harness Formats

**User Story:** As an artifact author, I want invalid format values to be caught at validation time with clear error messages, so that I can fix mistakes before building.

#### Acceptance Criteria

1. THE FrontmatterSchema SHALL define per-harness format validation that checks the `format` field against the valid values for each specific Harness.
2. WHEN a Harness_Config section contains a `format` field with a valid value for that Harness, THE parser SHALL accept the frontmatter without errors.
3. WHEN a Harness_Config section contains a `format` field with an invalid value, THE parser SHALL produce a validation error that identifies the offending Harness, the invalid value, and the list of valid values for that Harness.
4. THE Harness_Config sub-schema SHALL use `.passthrough()` to preserve existing harness-specific fields alongside the new `format` field.

### Requirement 9: Round-Trip Preservation

**User Story:** As a developer, I want the parse-then-serialize cycle to preserve per-harness format configuration, so that reading and writing an artifact does not lose format settings.

#### Acceptance Criteria

1. FOR ALL valid frontmatter containing `format` values in Harness_Config, parsing the frontmatter and then serializing it back SHALL produce YAML that, when parsed again, yields an equivalent frontmatter object.
2. THE parser SHALL preserve `format` values in the Harness_Config passthrough fields without stripping or transforming them.
3. THE parser SHALL preserve the deprecated top-level `type` field without modification during a parse-serialize cycle.
